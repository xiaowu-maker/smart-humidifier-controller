## 1. 设计原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 每个模块只负责一件事，修改一个功能只需改一个文件 |
| **依赖倒置** | 高层模块（Controller）不依赖低层模块的具体实现，依赖抽象接口 |
| **开闭原则** | 新增功能（如DHT22替代AHT20）不修改已有代码，只需新增实现 |
| **配置与代码分离** | 所有硬件引脚、阈值、WiFi凭证集中在 `config.h`，改配置不改代码 |
| **防御式编程** | 每个外部调用都有超时和重试机制，传感器失败不导致系统崩溃 |
| **可观测性** | 关键状态通过串口日志输出，运行时可诊断 |

## 3. 模块详细设计

### 3.1 SensorManager（传感器管理）

```
类名：SensorManager
文件：SensorManager.h / SensorManager.cpp
依赖：Adafruit_AHTX0 库, Wire (I2C)
```

**职责**：
- 初始化 AHT20 传感器
- 每 2 秒读取一次温湿度
- 检测传感器是否在线
- 提供缓存数据（防止读取失败时返回垃圾值）

**接口**：
```cpp
class SensorManager {
public:
    bool begin();                    // 初始化，返回true=成功
    bool readSensor();               // 执行一次读取
    float getTemperature();          // 获取最新温度（℃）
    float getHumidity();             // 获取最新湿度（%RH）
    bool isSensorOK();               // 传感器是否正常
    const char* getErrorMsg();       // 获取错误信息
private:
    float _temperature;
    float _humidity;
    bool _sensorOK;
    unsigned long _lastReadTime;
    String _errorMsg;
};
```

**错误处理**：
- 初始化失败 → `_sensorOK = false`，返回上次有效值
- 读取超时（>500ms）→ 重试1次，仍失败则标记异常
- 数值超范围（湿度0-100%，温度-40~80℃）→ 视为异常

## 5. 控制流

### 5.1 系统启动流程

```
上电 / 复位
    │
    ▼
Serial.begin(115200)
    │
    ▼
Logger: "System booting..."
    │
    ▼
I2C 总线初始化 (Wire.begin)
    │
    ├── 成功 ──▶ Sensor.begin()
    │               │
    │               ├── 成功 ──▶ Display.begin()
    │               │               │
    │               │               ├── 成功 ──▶ 进入主循环
    │               │               │
    │               │               └── 失败 ──▶ LED红色 + 蜂鸣器报警
    │               │                             系统仍运行(无显示模式)
    │               │
    │               └── 失败 ──▶ LED红色闪烁 + Logger错误
    │                             系统仍运行(仅显示错误)
    │
    └── 失败 ──▶ CRITICAL_ERROR（I2C总线故障，系统halt）
```

### 5.2 核心控制算法（滞回控制）

```
每次循环：
    读取湿度 H
        │
        ▼
    H 数据有效？
        ├── 否 ──▶ 保持当前状态 + 递增错误计数
        │          错误计数 >= 3 → SENSOR_ERROR 状态
        │
        └── 是 ──▶ 重置错误计数
        
        ▼
    ┌──────────────────────────────────────┐
    │         滞回控制逻辑 (Hysteresis)       │
    │                                      │
    │   H < humidityLow (45%) ?           │
    │       ├── 是 → Relay ON，状态=HEATING │
    │       │                               │
    │   H > humidityHigh (55%) ?          │
    │       ├── 是 → Relay OFF，状态=RUNNING│
    │       │                               │
    │   45% ≤ H ≤ 55% ?                   │
    │       └── 保持继电器当前状态(滞回区)    │
    │                                      │
    │   额外约束：                           │
    │   - 继电器状态变化后，最少维持 60 秒     │
    │   - 连续 3 次读数超阈值才动作（防抖）    │
    └──────────────────────────────────────┘
        │
        ▼
    更新 LED（HEATING→蓝, RUNNING→绿, ERROR→红）
        │
        ▼
    更新 OLED 显示
        │
        ▼
    delay(2000ms)
        │
        ▼
    下一轮循环
```

### 5.3 防抖策略

```
防止继电器频繁开关的3层保护：

第1层：滞回区间
  45%~55% 之间不改变继电器状态

第2层：采样防抖
  连续3次（6秒）读数都超过阈值才动作
  
第3层：最小运行时间
  继电器状态改变后，至少维持60秒不变
```

## 7. 错误处理策略

### 7.1 分级错误处理

| 级别 | 场景 | 响应 |
|------|------|------|
| **WARNING** | 单次传感器读取失败 | 使用缓存值，记录日志 |
| **ERROR** | 连续3次传感器失败 | LED红色，蜂鸣器间歇响，继电器保持当前状态 |
| **CRITICAL** | I2C总线故障 / 系统崩溃 | 所有输出安全态（继电器OFF），蜂鸣器长鸣 |

### 7.2 安全优先原则

> 任何不确定情况，继电器默认 **关闭**（Fail-Safe）

- 传感器故障 → 继电器关闭（防止过度加湿）
- WiFi断连 → 不影响本地自动控制（降级运行）
- 程序崩溃 → 看门狗自动重启
- 断电后恢复 → 从初始状态开始（继电器默认关闭）

## 8. 内存与性能预算

### 8.1 ESP32 内存预算

| 资源 | 预算 | 说明 |
|------|------|------|
| Flash | ~1MB / 4MB | 程序+SPIFFS |
| DRAM | ~100KB / 320KB | 运行时内存 |
| PSRAM | 4MB (如果型号支持) | 可选，Web页面缓存 |
| 栈深度 | 8KB main loop | 避免深层递归 |

### 8.2 时序预算

| 操作 | 耗时 | 频率 |
|------|------|------|
| AHT20 读取 | ~75ms | 每2秒 |
| OLED 刷新 | ~10ms | 每2秒 |
| 控制逻辑判断 | <1ms | 每2秒 |
| Web请求处理 | <50ms | 按需 |
| WiFi重连 | <5s | 断线时 |

**主循环总耗时**：~100ms，远小于 2000ms 循环间隔。

```
config.h 默认值 ← 运行时覆盖
    │
    ▼
保存到 SPIFFS（断电不丢失）
    │
    ▼
下次启动读取 SPIFFS → 覆盖默认值
```
