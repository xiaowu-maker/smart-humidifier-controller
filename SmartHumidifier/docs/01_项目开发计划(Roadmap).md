## 1. 项目概述

### 1.1 我们为什么要做这个项目

室内湿度是影响居住舒适度的关键因素。湿度过低（<30%）会导致皮肤干燥、咽喉不适、静电频发；湿度过高（>70%）则容易滋生霉菌、诱发呼吸道疾病。医学研究表明，人体最舒适的相对湿度范围是 **45% ~ 55%**。

市售智能加湿器价格从数百元到上千元不等，功能良莠不齐，且多数依赖云平台，存在隐私风险和断网失效的隐患。本项目旨在**用自己的双手打造一套低成本、高可靠性、完全本地可控的智能湿度管理系统**，让你在掌握嵌入式开发技能的同时，获得一台真正属于自己的智能设备。

### 1.2 项目目标

| 维度 | 目标 |
|------|------|
| 功能 | 自动将室内湿度维持在 45% ~ 55% |
| 成本 | BOM（物料清单）总成本控制在 80 元以内 |
| 可靠性 | 7×24 小时不间断运行，异常自动告警 |
| 扩展性 | 架构支持后续联网、手机控制、AI 分析等升级 |

### 1.3 核心技术栈

- **主控芯片**：ESP32 DevKit V1（Xtensa LX6 双核 240MHz，内置 WiFi + Bluetooth）
- **传感器**：AHT20（I2C 接口，精度 ±0.3°C / ±2%RH）
- **显示**：0.96 寸 OLED 128×64（SSD1306，I2C 接口）
- **执行器**：单路继电器（驱动加湿器）、三色 LED（状态指示）、蜂鸣器（告警）
- **输入**：2 个物理按钮（模式切换 / 设置）
- **开发框架**：PlatformIO + Arduino 框架
- **编程语言**：C++（嵌入式子集）

## 3. 版本演进总图

本项目采用**增量迭代**的开发模式，从最小可用产品（MVP）开始，逐步叠加功能。每一版都建立在前一版的基础上，且每一版都是一个**可运行、可验证**的完整系统。

```
V1.0 ──── MVP（基础功能版）
  │
  ├── 温湿度采集
  ├── OLED 显示
  └── 自动控制（开关加湿器）
  │
  ▼
V2.0 ──── 联网交互版
  │
  ├── WiFi 连接
  ├── Web 控制面板
  └── 手机端查看与控制
  │
  ▼
V3.0 ──── AI 分析版
  │
  ├── 湿度趋势预测
  ├── 自适应阈值调整
  └── 历史数据可视化
  │
  ▼
V4.0 ──── 产品化版
  │
  ├── 定制 PCB
  ├── 3D 打印外壳
  └── 固件 OTA 升级
  │
  ▼
V5.0 ──── 生态版（未来展望）
  │
  ├── 多房间联动
  ├── Home Assistant 集成
  └── 低功耗电池版本
```

**版本命名规则**：
- `V1.x`：核心功能迭代（传感器、显示、控制）
- `V2.x`：网络功能迭代（WiFi、Web、手机端）
- `V3.x`：智能化迭代（AI 分析、自适应）
- `V4.x`：产品化迭代（硬件定型、生产准备）

```cpp
// 第一个 ESP32 程序的核心代码
#include <Arduino.h>

void setup() {
    // 初始化串口，波特率 115200
    Serial.begin(115200);
    // 配置板载 LED（ESP32 DevKit 板载 LED 接 GPIO2）
    pinMode(2, OUTPUT);

    Serial.println("");
    Serial.println("===================================");
    Serial.println("  Smart Humidifier Controller v1.0");
    Serial.println("  Hello, Smart Humidifier!");
    Serial.println("===================================");
}

void loop() {
    // LED 闪烁：每秒变换状态
    digitalWrite(2, HIGH);
    delay(500);
    digitalWrite(2, LOW);
    delay(500);
}
```

### 6.6 常见问题与排查

| 问题现象 | 可能原因 | 解决方案 |
|----------|----------|----------|
| 烧录失败 "A fatal error occurred" | 串口被占用 / 驱动未装 | 关闭其它串口软件，检查 CP210x 驱动 |

### 7.5 AHT20 核心代码设计

```cpp
class SensorAHT20 {
public:
    // 传感器状态枚举
    enum Status {
        OK = 0,           // 正常
        ERROR_I2C,        // I2C 通信失败
        ERROR_NORESET,    // 复位失败
        ERROR_INVALID     // 数据无效
    };

    // 构造函数：传入 I2C 地址（默认 0x38）
    SensorAHT20(uint8_t addr = 0x38);

    // 初始化传感器（返回 true 表示成功）
    bool begin();

    // 触发一次测量并读取结果
    // 返回状态码，温湿度通过引用参数返回
    Status read(float &temperature, float &humidity);

    // 获取上一次读取的状态（用于快速检查）
    Status lastStatus() const;

private:
    uint8_t _addr;
    Status   _lastStatus;
    uint8_t  _rawData[6];   // 原始 6 字节数据缓存

    bool    _sendCommand(uint8_t cmd);
    bool    _readRawData();
    float   _calcTemperature(uint8_t *data);
    float   _calcHumidity(uint8_t *data);
};
```

### 7.6 数据校验规则

```
温度校验范围 : -10°C ～ +60°C
湿度校验范围 :   0% ～ 100%
连续异常上限 :   5 次（超过则触发蜂鸣器告警）
采样间隔     :   1 秒（根据实际需要可调整为 2 ~ 5 秒）
```

### 7.7 异常处理策略

1. **I2C 通信失败**：立即重试 3 次，每次间隔 10ms，仍失败则返回 `ERROR_I2C`
2. **数据越界**：丢弃本次数据，使用上一次有效值，计数器 +1
3. **连续异常达到阈值**：触发蜂鸣器告警，OLED 显示 "Sensor Error"
4. **传感器无响应超过 30 秒**：进入安全模式，关闭加湿器（防止失控）

### 7.8 里程碑验收标准

- 串口每秒输出温湿度数据，格式为 `[OK] Temp: 25.3°C, Hum: 52.7%`
- 用手捏住传感器，湿度读数明显上升；用嘴哈气，温度读数明显上升
- 拔掉传感器 I2C 线，系统在 3 秒内检测到异常并输出错误信息
- 连续运行 24 小时，数据稳定，无异常中断

### 9.5 滞回控制算法详解

滞回控制（Hysteresis Control）是工业上最常用的简单控制策略之一。其核心思想是**设置两个不同的阈值**来消除继电器在临界点附近的频繁切换。

```
算法逻辑：

湿度过低 ──→ 开启加湿器
    ↑              │
    │              │  湿度上升中
    │              ▼
    │     ┌────────────────┐
    │     │  滞回区间       │
    │     │  45% ←──→ 55%  │
    │     └────────────────┘
    │              │
    │              │  湿度下降中
    │              ▼
湿度过高 ──→ 关闭加湿器

具体规则：
  ● 当湿度 ≤ 45%  → 开启加湿器（无论之前状态如何）
  ● 当湿度 ≥ 55%  → 关闭加湿器（无论之前状态如何）
  ● 当 45% < 湿度 < 55% → 维持当前状态不变（滞回区间）

这样就避免了在 50% 临界点来回切换的问题。
```

**为什么使用滞回控制？**

如果没有滞回，当湿度恰好达到 50% 时，传感器噪声可能导致读数在 49.9% 和 50.1% 之间波动，继电器可能会在几秒内反复通断数十次，不仅损坏继电器触点，还会严重缩短加湿器寿命。滞回区间（45%~55%）提供了 10% 的缓冲区，即使传感器有 ±1% 的噪声，系统也能稳定运行。

### 9.6 安全保护机制

| 保护类型 | 触发条件 | 执行动作 |
|----------|----------|----------|
| 传感器故障保护 | 传感器连续 5 次读取失败 | 关闭加湿器，蜂鸣器告警，OLED 显示 "SAFE MODE" |
| 加湿器超时保护 | 持续运行超过 2 小时湿度无变化 | 关闭加湿器并告警，可能存在漏水或传感器故障 |
| 手动超驰 | 用户按下按钮切换到手动模式 | 停止自动控制，由用户手动控制开关 |
| 上电安全 | 系统上电启动 | 默认关闭加湿器，等待传感器数据稳定后再进入自动模式 |

### 12.5 趋势分析算法设计

```
算法名称：加权滑动窗口线性回归 (Weighted Sliding Window Linear Regression)

原理：
  对最近 N 个湿度数据点进行线性回归拟合，计算趋势斜率。
  斜率为正 → 湿度上升中
  斜率为负 → 湿度下降中
  斜率的绝对值越大 → 变化越剧烈

参数配置：
  - 窗口大小 N = 60（最近 60 分钟的数据）
  - 权重策略：近期数据权重更高（指数衰减）
  - 预测范围：未来 1 小时
  - 更新频率：每 5 分钟重新计算一次

自适应阈值调整规则：
  - 如果预测 30 分钟后湿度会低于 45%，提前开启加湿器
  - 如果预测 30 分钟后湿度会高于 55%，提前关闭加湿器
  - 如果环境稳定（湿度变化 < ±1%/h），保持标准阈值
  - 如果环境波动剧烈（湿度变化 > ±5%/h），适当放宽滞回区间至 40%~60%
```

## 14. 附录：常见问题与建议

### 14.1 给初学者的建议

1. **不要贪快**：每个 Phase 都按顺序完成，不要跳过。前面任何一个环节没搞扎实，后面都会加倍还回来。
2. **多用串口调试**：串口是你最好的朋友。遇到任何问题，第一时间加 `Serial.println()` 输出信息。
3. **备份你的代码**：每次完成一个功能就 commit 一次。不要等到全部写完再 commit。
4. **善用搜索引擎**：80% 的问题别人都遇到过。关键词：`ESP32` + `你的问题` + `Arduino`。
5. **准备一个笔记本**：记录每次遇到的错误和解决方案。这会成为你最宝贵的技术积累。

### 14.2 常见问题速查

| 问题 | 排查思路 |
|------|----------|
| 程序烧录失败 | 检查串口驱动、选对板型、按住 BOOT 键再烧录 |
| 传感器读数固定不变 | 检查 I2C 接线是否松动、地址是否正确 |
| OLED 不显示 | 检查 I2C 地址（0x3C vs 0x3D）、检查电源电压 |
| 继电器不动作 | 检查 GPIO 电平逻辑（高/低电平触发）、检查驱动三极管 |
| WiFi 频繁断连 | 检查 3.3V 供电是否足够、天线附近不要有金属遮挡 |
| Web 页面无法打开 | 确认手机/电脑与 ESP32 在同一局域网、检查防火墙 |
