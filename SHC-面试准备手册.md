## 二、面试官可能会问的问题

### Q1：为什么选择 ESP32 而不是 Arduino Uno？

**回答要点**：
- ESP32 有 WiFi + Bluetooth（为 V2.0 联网功能预留）
- 240MHz 双核 vs Uno 的 16MHz 单核
- 320KB RAM vs Uno 的 2KB——足够跑 Web Server
- 价格差不多（ESP32 ¥25 vs Uno ¥20）
- "用 Uno 也能做 V1.0，但 V2.0 必须换芯片——不如一步到位"

### Q2：为什么不直接用 delay()？为什么要用 millis()？

**回答要点**：
- delay() 期间 CPU 完全空转，不能处理任何事情
- 如果用 delay(2000) 等传感器，这 2 秒内按钮按了也没反应
- millis() 非阻塞：让单核 CPU "看起来"同时做多件事
- 这是嵌入式开发的基本功——"状态机 + 定时器"思想

### Q3：滞回控制是什么？为什么要用？

**回答要点**：
- 如果只有一个阈值（50%），湿度在 49.9%~50.1% 波动时继电器会疯狂开关
- 设两个阈值（45% 和 55%），中间 10% 是"死区"
- 这是工业控制的经典做法（空调、冰箱都这么干）
- 再加三重保护：滞回 + 最短运行时间 + 最短关闭时间

### Q4：传感器坏了怎么办？

**回答要点**：
- 不会崩溃——采用"优雅降级"策略
- 前几次失败：使用缓存数据 + 记录错误
- 连续 5 次失败：判定传感器故障，LED 变红、蜂鸣器告警、继电器强制关闭
- 传感器恢复后：自动切回正常模式
- 这是 Fail-Safe 设计原则

### Q5：代码结构是怎么组织的？

**回答要点**：
- 四层架构：驱动层 → 硬件抽象层 → 工具层 → 应用层
- 每层只做自己的事，上层不依赖下层实现细节
- 7 个模块、每个对应一个 C++ 类
- 如果换一个传感器（如 SHT30），只改 SensorManager，其他模块不受影响

### Q6：这个项目最难的部分是什么？

**回答要点（任选一个展开）**：

1. **架构设计**：怎么把 7 个模块拆得合理，不耦合
2. **非阻塞编程**：所有模块用 millis() 协同，没有一处 delay()
3. **LED 颜色宏冲突**（Bug #4）：`#define WHITE` 污染了 `LedColor::WHITE` 枚举，学到了 C++ 预处理器的工作原理
4. **从零建立工程规范**：PRD、架构文档、测试手册、Bug 手册——不是写完代码就完事

### Q7：如果重新做，哪里可以改进？

**回答要点**：
- 传感器数据验证可以更完善（加上卡尔曼滤波）
- 可以用 FreeRTOS 任务代替手动 millis() 调度（ESP32 支持）
- 可以加单元测试框架（Unity Test）
- 但这都是 V2.0/V3.0 的事——"先跑起来，再跑好"

## 五、如果面试官让你现场写代码

准备好现场在白板/编辑器里写出以下任意一段：

### 滞回控制逻辑（核心算法）
```cpp
if (humidity < 45.0) {
    relay.on();    // 湿度太低，开加湿器
} else if (humidity > 55.0) {
    relay.off();   // 湿度够了，关加湿器
}
// 45%~55% 之间：什么都不做（滞回区）
```

### 非阻塞定时（millis 模式）
```cpp
static unsigned long lastRead = 0;
if (millis() - lastRead >= 2000) {  // 每 2 秒
    lastRead = millis();
    readSensor();  // 做你的事
}
// 没有 delay()，循环继续执行其他任务
```

### 按钮消抖
```cpp
if (reading != lastReading) {
    lastDebounceTime = millis();  // 重置消抖计时器
}
if ((millis() - lastDebounceTime) > 30) {
    // 30ms 内电平没变 → 确认有效
    if (reading != debounced) {
        debounced = reading;
        // 触发事件...
    }
}
```
